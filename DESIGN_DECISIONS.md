## D-001. 영업일 시작 03:00 KST

**결정** : 매장의 영업일은 03:00 KST에 시작하여 다음날 02:59:59에 종료.

**사유** :
- 새벽 매장(주점·치킨·24시간) 대응. 새벽 1시 매출이 '오늘'로 잡혀야 사장님 멘탈 모델에 맞음.
- 단순 calendar day (00:00 ~ 23:59) 사용 시 새벽 매출이 별도일로 떨어져 분석 왜곡.
- 03:00이라는 시각은 한국 새벽 매장의 일반적 마감 시각.

**대안 검토** :
- ✗ 단순 calendar day : 새벽 매장 대응 안 됨
- ✗ 매장별 자유 설정 : 본사 정산·집계 복잡도 ↑. 1차 범위 외.

## D-002. 환불은 원거래일 매출에서 차감

**결정** : 환불 발생 시 매출 집계는 **원거래(결제)일의 영업일**에서 차감.

**사유** :
- 세무·회계 표준 관행. 원거래일 기준으로 매출 신고하므로 일치 필요.
- 일별 매출 통계의 정확성. 5/15에 결제하고 5/20 환불 시 5/20 매출을 음수로 잡으면 5/20 통계가 왜곡됨.

**부작용 / 대응** :
- 이미 정산 완료된 일자의 매출이 사후 환불로 줄어들 수 있음 → 다음 정산 회차에서 차감 (`STATES.md` §8.2 참조)
- 사장님 멘탈 모델 : "어제 환불이 어제 매출을 줄였다"는 표시가 직관적. 매출 한눈에 보기에서 환불 별도 라벨로 노출.

**대안 검토** :
- ✗ 환불일에 음수 매출 기록 : 일별 통계 왜곡

## D-003. effectiveStatus 우선순위

**결정** : `수동 품절 > 자동 품절 > 숨김 > 판매 시간 외 > 판매중` 순으로 평가.

**사유** :
- "사장님 명시적 의사 > 자동 시스템 판정" 원칙
- '오늘 품절'은 사장님이 의도적으로 막은 상태이므로 가장 강함
- 자동 품절(재고 0)은 시스템 자동 판정. 수동 품절보다 약함
- 숨김은 메뉴 자체를 손님에게 안 보이게 하는 영구 설정. 판매 시간보다 강함
- 판매 시간 외는 시간대 조건이므로 가장 약함

## D-004. status enum에서 'soldout' 제거

**결정** : `Menu.status` enum에서 `soldout` 제거. `manual_sold_out` bool로 분리.

**사유** :
- 기존 mockup에는 `status = '판매중' / '품절' / '숨김'` 셋이었음
- 품절은 영업일 단위 토글이라 영구 설정인 status와 의미가 다름
- 분리하면 영업일 마감 시 자동 리셋(품절 해제) 로직이 단순해짐

**마이그레이션** : 기존 `status='soldout'` 데이터는 `manual_sold_out=true` + `status='available'` 로 변환.

---

## D-005. 영업일 마감 시 prepared 유지, manual_sold_out 해제

**결정** : 영업일 시작 시 자동 처리는 다음과 같이 비대칭.
- `StockItem.prepared` : **유지** (사장님이 매일 직접 갱신)
- `StockItem.remaining` : `prepared` 값으로 리셋
- `Menu.manual_sold_out` : **모두 false로 자동 해제**

**사유** :
- `prepared` 자동 변경(예: 어제 값 복제)은 사장님 의도와 다를 수 있어 위험. 유지가 안전.
- `manual_sold_out`은 이름 자체가 '오늘 품절'. 다음 영업일에는 당연히 해제되어야 의미가 맞음.

**부작용** : 사장님이 매일 아침 준비량을 확인·갱신해야 함. 이를 돕기 위해 영업일 시작 시 어제 통계(yesterday_prep / yesterday_sold)를 카드에 표시.

## D-006. 부가세 산식과 사업자 유형 처리

**결정** : 일반과세 매장은 `vat = round(taxable / 11)`. 간이과세는 1차 릴리즈에서 `vat=0` 으로 표시하고 정산에서 별도 처리. 면세 매장은 1차 범위 외.

**사유** :
- 1차 릴리즈의 주요 타겟이 일반과세 소상공인
- 간이과세는 매출 1억 미만 영세 사업자 대상. 부가세 신고 방식이 다르므로 어드민에는 vat=0 표시가 혼란 적음.
- 면세(농산물·일부 식품)는 메뉴별 분기 로직 필요. 1차에 포함하면 복잡도 ↑

## D-007. 시간대 매출 hybrid 출처

**결정** : 당일 시간대 매출은 결제 raw에서 GROUP BY (실시간), 과거 시간대 매출은 사전 집계 테이블에서 조회.

**사유** :
- 당일 분석은 실시간성이 중요 (1시간 전 매출도 보고 싶음)
- 과거 분석은 응답 속도가 중요 (한 달치 시간대별 평균을 매번 raw 조회하면 느림)
- 영업일 마감 배치에서 시간대별 사전 집계 테이블 적재

---

## D-008. 직원호출 통계 기준 = 당일 영업일

**결정** : 직원호출 메시지의 'usage(호출 횟수)' / '평균 처리 시간' 은 **당일 영업일** 기준으로 표시.

**사유** :
- 사장님은 '오늘 운영 상황' 파악이 주 목적. 누적 카운트는 통찰이 약함.
- 매일 0으로 리셋되므로 메시지 간 비교가 일관됨.
- 7일 평균 등 추가 인사이트는 옆에 보조 표시.

**구현** : `StaffCallEvent.called_at` 의 영업일이 오늘인 레코드만 집계. 사전 집계 없이 raw 조회로도 부하 적음 (매장당 일 100건 미만 예상).

---

## D-009. 직원호출 '처리 완료' = 명시적 버튼 클릭

**결정** : 호출 처리 완료 트리거는 직원이 PC 에이전트 팝업에서 명시적 '완료' 버튼 클릭.

**사유** :
- 팝업 닫기는 부정확 (실수로 닫음, X 버튼 클릭 등)
- 자동 만료는 평균 처리 시간 통계 왜곡
- 명시적 버튼은 직원 행동 1회 추가 필요하지만 정확도 ↑

**미처리 처리** : 10분 경과 시 사장님 알림 + 빨간 배지. 영업일 마감 시 자동 만료(통계 제외).

---

## D-010. 동시 호출 디바운스 30초

**결정** : 같은 테이블에서 같은 메시지를 30초 안에 다시 누르면 무시.

**사유** :
- 손님이 답답해서 여러 번 누르는 패턴 발생 가능. 직원 입장에선 노이즈.
- 30초는 직원이 응답하기에는 짧고, 손님이 '잘못 눌렀나' 의심하기 직전.

**예외** : 다른 메시지는 즉시 호출 가능. 다른 테이블도 즉시 호출 가능.

---

## D-011. 정산 D+1 출금

**결정** : 일 정산. 영업일 종료 후 다음날 D+1에 출금.

**사유** :
- 소상공인 현금 유동성 중요
- 주 단위는 자영업자 부담 ↑

**예외** : 환불로 인한 차감액이 다음 회차 정산액보다 클 경우 `held` 상태로 보류. CS 채널 후속 처리.

## D-012. 채널 변경 시 사전 안내 모달

**결정** : `OrderChannelConfig.mode` 변경 시 사전 안내 모달 강제 노출. 사장님 명시적 확인 후에만 변경 가능.

**사유** :
- 채널 변경은 손님 화면을 즉시 바꾸는 큰 영향
- 실수로 매장 → 포장만 변경하면 영업 중단됨
- 모달은 영향(테이블 QR/포장 QR 노출 여부)을 명시적으로 보여 줘 결정 도움

**저장 시점 검증** :
- `mode=takeout` 변경 시 미수락 포장 예약이 있으면 차단

## D-013. 메뉴 가격 30% 변경 시 확인

**결정** : 메뉴 가격을 한 번에 30% 이상 변경하려 할 때 추가 확인 모달.

**사유** :
- 오타 방지 (5,000원 → 50,000원 같은 실수)
- 30% 임계값은 일반적 정상 인상폭의 상한

---

## D-014. 옵션 항목 가격 매장/포장 단일

**결정** : 옵션 그룹의 항목(예: 샷 추가 +500원)은 매장/포장에 따라 다른 가격을 가질 수 없음.

**사유** :
- 매장가/포장가 분리는 메뉴 수준에서만 (단일 옵션이 두 가격을 갖는 것은 사용 사례 적음)
- 운영 복잡도 ↓

**확장 가능성** : 필요 시 OptionItem에 `takeout_add_price` 추가하면 됨. 1차 범위 외.

---

## D-015. 일괄 삭제는 Soft Delete

**결정** : 메뉴·옵션·세트 등 마스터 데이터 삭제는 Soft Delete (DB 보존 + `deleted_at` 표시).

**사유** :
- 진행 중 주문에 묶인 메뉴를 hard delete 하면 주문 표시 깨짐
- 매출 분석에서 과거 메뉴 데이터 참조 필요
- 사장님이 실수로 삭제해도 본사 CS 채널로 복원 가능

**구현** : 모든 마스터 엔터티에 `deleted_at timestamptz NL` 추가. 손님 화면은 `deleted_at IS NULL` 필터.

---

## D-016. 50~60대 사장님 UX 우선

**결정** : 폰트 크기·터치 영역·대비를 50~60대 사용자 기준으로 설계. 보조 설명문 적극 사용.

**사유** :
- 소상공인 사장님 평균 연령대
- 노안·시력 저하 고려 (12px 이하 본문 텍스트 금지)
- 친근·존중 톤. 공포 표현 회피.

**구체** :
- 본문 텍스트 최소 13px
- 터치 영역 44x44px 이상
- 대비 WCAG AA 이상
- 보조 설명문은 항상 13px, 회색 (`--ink-3`)

## D-017. 손님 화면 = 별도 서비스

**결정** : 사장님 어드민(`index.html`)과 손님 주문 화면(`order.qrorder.co.kr/...`)은 **별도 프로젝트**.

**사유** :
- 사용자 / 기능 완전 다름
- 손님 화면은 비로그인 + 모바일 최적화
- 어드민은 데스크탑 PC 최적화 + 로그인

**연결** : 데이터 모델 공유. 어드민에서 변경한 메뉴 / 가격 / 재고 / 채널 설정은 손님 화면에 실시간 반영.

**손님 화면 명세** : 본 문서 범위 외. 별도 명세서 작성 필요.

## D-018. 영업시간과 영업일은 다름

**결정** : 매장 영업시간(예: 11:00 ~ 22:00)과 영업일(03:00 ~ 02:59)을 분리.

**사유** :
- 영업시간 : 손님 주문 가능 시간
- 영업일 : 매출 집계·재고 차감의 시간 단위
- 새벽 매장은 영업시간이 영업일을 넘나듦 (예: 18:00 ~ 02:00은 같은 영업일)

**충돌 처리** : 영업시간 외에는 손님 주문 차단. 영업일 마감(02:59:59) 직후 영업시간 내라도 새 영업일 시작.
