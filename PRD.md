# QR오더 사장님 관리자 — 매출·정산 영역 개편안

> 비개발 사장님이 "오늘 얼마 벌었고, 언제 얼마 들어와요?"를 한 번에 보고 이해할 수 있게 만드는 게 목표예요.

- **작성일**: 2026-05-20
- **대상**: 점주(사장님) 단일 권한
- **범위**: 주문 내역 · 매출 한눈에 보기 · 매출 상세 보기 · 정산 달력 · 정산 내역 · 정산 계좌

---

## 1. 한 줄 요약

> 주문(들어온 돈) → 매출(쌓인 돈) → 정산(받을 돈) 흐름을 **일자 단위**로 일관되게 보여주고, 사장님이 헷갈리지 않게 용어를 사람의 말로 정리해요.

## 2. 배경과 문제

### 기존 화면이 헷갈렸던 이유

1. **숫자가 화면마다 달랐어요**
   - 매출 대시보드 17.5M / 결제수단별 29.2M / 메뉴별 22.5M / 수수료 페이지 26.3M
   - 한 사장님이 "이번 달 매출 얼마야?"라고 물으면 답이 4개

2. **정산 회차 ID가 중복 생성됐어요**
   - 토·일 매출이 같은 정산일로 묶이면서 같은 ID 3건 발생
   - 회차 클릭 시 첫 회차로만 이동 (나머지 영원히 접근 불가)

3. **카드사 영업일 묶음을 "5/1~5/3"처럼 표시**
   - 사장님: "왜 승인일이 3일치로 묶여있지?"

4. **수수료율 정산식과 표가 달랐어요**
   - 정산식: 모든 거래에 일률 2.0%
   - 수수료 페이지: 카드 국내 2.0% / 해외 3.5% / 카카오 2.3% 등 차등
   - → 정산 금액 신뢰 불가

| 매출 상세 보기 - 시간대별 | 시간대 (24h) | 오늘 vs 4주 동요일 평균, 시간대 필터 |

## 5. 데이터 모델 (단일 진실의 원천)

### 📌 진실의 원천: `DAY` 배열 (15일치 일별 매출)

```js
DAY = [{ d:'5/15', w:'금', gross:1234500, disc:12000, pt:22000, ref:34000, rcnt:2, ocnt:46 }, ...]
```

- 모든 화면의 매출은 DAY에서 파생 (METHOD / MENU / CATEGORY / CHANNEL 합계도 DAY 15일 합 = 17,533,100원에 맞춤)
- **결과: 화면별 매출 숫자가 모두 일치**

### SETTLEMENTS (정산 회차)

DAY에서 자동 생성. 핵심 룰:
- 영업일 D+2 (한국 카드사 표준)
- 주말·공휴일 스킵 (KST 자정 기준 → `toISODate` 헬퍼로 UTC 변환 버그 회피)
- 토·일·공휴일 매출은 다음 영업일 회차로 자동 합산
- ID 포맷: `STL-{정산일YYYYMMDD}-{영업일MMDD}` (고유 보장)

### ORDERS (주문)

- date(결제일) 기준으로 `findSettlementByOrderDate()`로 정산 회차 자동 매핑
- `o.stl` 표시 라벨은 SETTLEMENTS에서 동적 파생 (수동 입력 폐지)
- `o._settlementId`로 정산 회차 클릭 시 정확한 회차로 이동

## 6. 비즈니스 룰

### 정산 금액 계산

```
수수료 기본    = (거래 - 환불) × 2.0%        ← 결제수단별 가중평균
부가세        = 수수료 기본 × 10%
수수료 합계   = 수수료 기본 + 부가세

쿠폰 차감(가맹점 발행)  = 거래 × 0.5% (쿠폰 사용일에만)
쿠폰 보전(테이블 발행)  = 거래 × 1.2% (쿠폰 사용일에만)

거래 정산금   = 거래 - 환불 - 수수료 합계 - 쿠폰 차감
최종 정산금   = 거래 정산금 + 쿠폰 보전
```

### 영업일 마감 시각

- 모든 영업일: **03:00 ~ 익일 02:59 (KST)**
- 자정 이후 새벽 결제 → 같은 영업일 회차에 포함
- 정산 상세 표에 영역 구분 행 삽입: `⏰ 자정 이후 · 5/15(금) 영업일 마감(5/16 03:00) 전까지 동일 회차에 포함`

### 카드번호 마스킹 (PCI DSS 준수)

- 1년 이내 거래: `BIN(앞 6자리) + 끝 4자리` 노출
- 1년 초과 거래: 전체 마스킹 `****-****-****-****` + 검색 차단

### 정산 사이클별 처리

| 결제수단 | 정산 방식 |
|----------|-----------|
| 카드 (국내) | D+2 영업일, 수수료 2.0% |
| 카드 (해외) | D+2 영업일, 수수료 3.5% |
| 카카오페이 | D+2 영업일, 수수료 2.3% |
| 네이버페이 | D+2 영업일, 수수료 2.4% |
| 토스페이 | D+2 영업일, 수수료 2.3% |
| 계좌이체 | D+2 영업일, 수수료 1.5% |
| 현금 | 정산 사이클 없음 (즉시 매장 보관) |
| 자체 포인트 / 상품권 | 별도 처리 (정산 없음) |

## 7. 사용자 흐름 시나리오

### A. "오늘 얼마 받을지" 확인 흐름

```
사이드바 → 정산내역 조회
   ↓ (기본 진입: 오늘 1일치)
"5/15(금) · 정산일 5/19(화) · 46건 · 정산금 1,182,730원" 노출
   ↓ 행 클릭
정산 회차 상세 (STL-20260519-0515)
  - 받으실 금액 1,182,730원
  - 거래·건수·수수료 3분할
  - 결제 내역 8건 (자정 이후 새벽 결제 영역 구분)
```

### B. "이번 달 정산 총합" 확인

```
정산내역 조회 → 칩 [이번 달] 클릭
   ↓
"이번 달 (5월) · 15일치 · 거래 17,533,100원 · 정산금 16,859,714원"
```

### C. 특정 주문 → 정산 회차 추적

```
주문 내역 → 주문 클릭 (예: #20260515-0046)
   ↓ 모달
"정산 회차: STL-20260519-0515 · 5/19(화) 정산대기 ›" 클릭
   ↓ 자동 이동
해당 회차 상세 페이지
```

### D. 정산 달력에서 일자 클릭

```
정산 달력 → 5/19 셀 클릭 (정산일 기준)
   ↓ (중간 정산내역 조회 단계 생략)
정산 회차 상세로 직접 진입
```

## 8. 사장님이 알아둘 안내

### 영업일이 새벽을 거치는 매장

- 카드사 영업일은 **03:00 마감 고정**
- 매장 영업시간(예: 호프집 18:00~04:00)이 새벽을 거치면 사장님 기준 "어제 매출"과 카드사 영업일이 한두 시간 어긋날 수 있음
- 현재는 안내 배너 없이 표 안의 "자정 이후 새벽" 영역 구분으로 표현 (사용자 요청)

### 정산 회차 ID 표시 정책

- 정산내역 조회 표: ID 미노출 (직관성 우선)
- 정산 회차 상세: ID 노출 (정산 1건 추적용 식별자)
- 주문 모달: 해당 주문이 속한 회차 ID로 이동 링크
