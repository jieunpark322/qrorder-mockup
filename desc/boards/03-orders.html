📋주문 내역 주문 관리 orders
📍 핀 5개
1상단 바 (실시간 · 집계 기준 · 내보내기)
기능·목적
  • 화면 최상단 운영 바. 실시간 주문 유입 알림, 매출 집계 기준 전환, 현재 조회 결과의 엑셀 내보내기, 카드번호 표시 정책 안내를 한 곳에 모아 운영자가 화면을 떠나지 않고 핵심 운영을 제어하게 한다.
  • 왜 있는가: 주문 내역은 영업 중 가장 자주 보는 화면이므로, 신규 주문을 놓치지 않게 하고(실시간) 사장님이 보는 수치의 '집계 기준'을 명확히 고정하기 위함.
동작·상태값
  • 🟢 실시간 배지(주문 내역 제목 옆): 항상 표시되는 상태 표식. 신규 주문 도착 시 PC 에이전트 팝업+사운드, 모바일 앱 푸시로 알린다(SPEC §1.8). 배지 자체는 클릭 동작 없음.
  • 집계 기준 세그먼트 토글 🕛 자정 기준 ↔ 🕒 영업시간 기준: 클릭 시 state.useBusinessHours false↔true 전환 후 즉시 renderApp(). 기본값=자정 기준(false). 매장이 영업시간 기준으로 바꾸면 그 선택을 매장별로 저장(라스트 메모리)해 다음 진입·이후에도 같은 기준으로 보여준다. '영업시간 기준' 선택 시 라벨에 (open~close) 시각을 함께 노출하고, 본문 상단에 안내 배너(bhActiveNotice())를 띄운다.
  • ⚙️ 영업시간 설정 버튼: openBusinessHoursModal() 호출 — 사장님이 영업 시작/마감 시각을 직접 변경(자정 넘김 매장 지원).
  • 📤 내보내기 버튼(.btn-primary): openExport(rows.length, '주문 내역', 기간라벨) 호출. 현재 필터/정렬이 적용된 조회 결과 전체(rows.length 건)를 xlsx로 내보낸다(표시 개수 페이징과 무관).
  • ⓘ 카드번호 표시 정책(목록 헤더): 클릭 시 마스킹 정책 alert 노출.
데이터·연동
  • 데이터 소스: ORDERS 주문 트랜잭션 실시간 조회. 신규 주문은 OrderStatus.pending→accepted (PG 승인 콜백, Payment.status='paid') 시점에 목록 진입(STATES §1.1).
  • 집계 기준 플래그: useBusinessHours:boolean, businessHoursOpen/Close:'HH:MM'. 영업일 경계는 03:00 시작·다음 날 02:59 종료이며 자정 이후 거래도 영업 시작일 매출로 묶인다(SPEC §1.1).
  • 내보내기 파라미터: 건수, 화면명, 기간 라벨(today/yesterday/week/month/3m/custom 분기). 출력 범위 = 서버 측 필터 쿼리 결과 전체.
  • 실시간 갱신: 신규 주문 도착 시 전체 새로고침 없이 데이터만 재조회(SPEC §2.5). 알림 채널은 PC 에이전트/앱 푸시(SPEC §1.8).
예외처리·UX문구
  • 권한: 조리 시작/완료·픽업·예약 수락/거절 등 상태 변경은 staff도 가능하나, 환불(결제 취소) 및 주문 거절은 owner 전용(SPEC §1.6, STATES §1.4). staff도 주문 내역은 전체 기간 조회 가능(사장님과 동일).
  • 영업시간 기준 ON + 자정 넘김 매장: 강조 안내 배너 노출 — "영업시간 기준 (open ~ close)으로 보고 있어요" / "영업 시작 시각부터 다음 날 마감 시각까지를 하나의 영업분으로 묶어서 보여드려요. 자정 이후 거래도 영업 시작일의 매출로 포함됩니다."
  • 영업시간 기준 ON + 자정 안 넘김 매장: 가벼운 한 줄 안내 — "🕒 영업시간 기준 (open ~ close)으로 보고 있어요."
  • 내보낼 데이터가 0건이면 내보내기 버튼은 노출하되 빈 결과 안내로 막는다(능동 톤).
2주문 요약 KPI
기능·목적
  • 현재 조회 범위(집계 기준+검색 필터 반영)의 주문 현황을 총 주문 · 진행중 · 완료 · 취소 · 매출 합계 5개 카드로 요약한다.
  • 상태별 카드는 단순 요약이 아니라 상태 필터 진입점으로도 동작해, 한 번에 해당 상태 목록으로 좁혀 볼 수 있게 한다.
동작·상태값
  • 5개 카드 그리드(repeat(5,1fr)): ①총 주문 ②진행중 ③완료 ④취소 ⑤매출 합계.
  • 총 주문/진행중/완료/취소 카드 클릭 → setOrderStatus('all'|'진행중'|'완료'|'cancel-all'). 이미 선택된 카드를 다시 누르면 'all'로 토글 해제. 선택 카드는 색 테두리+box-shadow로 강조(진행중=amber, 완료=blue, 취소=red).
  • 매출 합계 카드: 완료 기준으로만 합산(status==='완료'인 주문의 amt 합), 클릭 진입점 없음. 하단에 "완료 기준" 라벨 고정.
  • 진행중 카드: 건수>0이면 "처리 대기" 보조 라벨을 amber로 노출.
  • KPI는 상태 필터 적용 전 기준으로 산출(카드가 진입점 역할을 하므로 항상 전체 상태 수치를 보여줌). 단 기간·채널·결제수단·메뉴·금액·번호 검색은 반영된 수치.
데이터·연동
  • 집계 식: inProgressCnt=count(status='진행중'), completedCnt=count(status='완료'), cancelAllCnt=count(status='취소'), totalCnt=rows.length.
  • totalAmt = Σ(o.amt where status='완료') = 완료 결제금액 합계(결제 기준) — 매출 화면의 실매출(actual=gross−할인−포인트−환불)과는 다른 개념이라 라벨은 '완료 결제 합계' 권장. 취소/환불 건 제외, 환불은 원거래일 매출에서 차감(SPEC §1.5).
  • 표시 단위: 건수는 '건', 매출은 '원'. 천단위 콤마(fmt()).
  • 필드: status:'진행중'|'완료'|'취소'(표시값), amt:number 결제금액.
예외처리·UX문구
  • 해당 기간 주문 0건이면 모든 카드 0 표시, 목록은 빈 상태 안내로 전환.
  • 취소 0건이면 취소 카드 숫자를 회색(--ink-3)으로 톤다운, >0이면 red 강조.
  • 선택된 상태 필터는 카드 강조와 '더 자세히 찾기' 패널 요약 라벨에 동시 반영("전체 상태/진행중/완료/취소·환불").
3더 자세히 찾기 (고급 검색)
기능·목적
  • 기간·상태·메뉴·채널·결제수단·금액 범위·주문번호/승인번호/결제번호를 한 패널에서 조합해 정밀 검색한다.
  • 여러 조건을 초안(draft)으로 모아 두고 [검색] 버튼으로 한 번에 적용하는 방식이라, 조건마다 화면이 튀지 않고 의도한 조합으로만 조회된다.
동작·상태값
  • 펼침/접힘 토글(toggleOrderAdvanced()): 펼칠 때 현재 라이브 필터를 orderFilterDraft로 복사. 헤더에 적용 중 조건 개수 배지("N개 조건 적용 중")와 현재 요약("기간 · 상태 …") 노출. 토글 라벨 펼치기/접기.
  • 기간 칩: 오늘(5/15)/어제/이번 주/이번 달/최근 3개월/📅 기간 직접 선택. 기본 기간=오늘. 직접 선택은 openDateRangePicker('orders')로 시작·종료일 지정.
  • 상태 칩: 전체/진행중/완료/취소·환불 전체. 메뉴 입력: 요약 라벨(o.menu)과 상세 items.name까지 부분 일치. 채널 칩: 전체/먹고가기/포장하기/(포장 예약은 기능 활성 시만 노출). 결제수단 칩: 전체/카드/카카오페이/네이버페이/토스페이/현금. 금액 범위: 최소~최대 입력. 주문/승인번호: 주문번호·승인번호·결제번호(tid) 통합 입력.
  • 하단 버튼: [조건 비우기](clearOrderFilterDraft() — draft만 기본값 초기화) / [🔍 검색](applyOrderFilters() — draft를 라이브 state로 일괄 복사 후 갱신).
  • 조건 적용 시 목록·KPI·결과 건수 즉시 갱신, 페이지는 1로.
데이터·연동
  • 검색은 부분 일치·대소문자 무시(SPEC §2.6). 번호 검색은 id + tid + makeApproval(o) 결합 문자열 대상.
  • 필드: orderDateFilter(today/yesterday/week/month/3m/custom/all), orderStatusFilter(all/진행중/완료/cancel-all), orderChannelFilter(all/매장/포장/포장 예약), orderPayFilter(payMain(o) 1depth 매칭), orderMenuSearch:string, orderAmountMin/Max:number, orderSearch:string.
  • 조회 정책: 한 번에 최대 3개월(90일), 데이터 1년 보관 후 자동 삭제. custom 종료일 max는 시작일 기준 90일로 동적 제한, 시작일 < 2025-05-15 차단.
  • 페이징: 페이지당 10/20/50/100 (기본 20), 이전/다음+번호 페이징(SPEC §2.6).
예외처리·UX문구
  • 결과 0건: "조건에 맞는 주문이 없어요. 날짜 범위를 넓히거나 필터를 해제해 보세요." (능동 톤).
  • 패널 안내 카드: "💡 여러 조건을 함께 골라 두고 아래 검색 버튼을 눌러야 적용돼요." / 조회 정책 안내 "🗓️ 한 번에 최대 3개월 조회 가능 · 데이터는 1년 보관 후 자동 삭제".
  • 기간 90일 초과: "주문 내역은 한 번에 최대 3개월(90일)까지만 조회할 수 있어요. 기간을 줄여서 다시 선택해 주세요."
  • 1년 이전: "1년 이전 데이터는 보관 정책에 따라 조회할 수 없어요. (보관 기간: 1년)". 종료일<시작일: "종료일을 시작일 이후로 선택해 주세요". 미래일: "오늘까지의 날짜에서 선택해 주세요".
  • staff도 기간 칩 전체 사용 가능 — 전체 기간 조회(사장님과 동일).
4주문 목록 표 · "자세히" → 주문 상세 모달
기능·목적
  • 조회된 주문을 표로 보여주는 본문. 컬럼: No. · 주문일시 · 주문번호 · 채널 · 메뉴 요약 · 결제 시점 · 결제수단 · 결제금액 · 상태 · 자세히.
  • 행 전체 또는 우측 "자세히 ›" 클릭 → openOrderDetail(o)로 주문 상세 모달(영수증 단위) 오픈. 운영자가 한 주문의 전체 영수증·결제·환불 내역을 확인하고 취소/영수증 출력을 수행하는 진입점.
동작·상태값
  • 헤더 클릭 정렬: 주문일시/주문번호/채널/메뉴/결제수단/결제금액/상태 컬럼에 ▲▼ 토글(setOrderSort, applySort). 결제 시점/No./자세히 컬럼은 정렬 없음.
  • 채널 셀: 먹고가기(테이블 N)/포장하기/포장 예약 표시(o.channel — "포장 예약"은 channel=takeout + 예약 속성(pickup_time·reservation_accepted_at)의 표시 파생 라벨, DATA_MODEL §4.1). 결제 시점 셀: 선불/후불 pill(orderPayType(o) — 포장·예약=선불, 매장 17~21시=후불 데모 규칙). 결제수단 셀: payLabel+paySub, 미결제는 "미결제".
  • 결제금액 셀: 정상은 amt 굵게; 취소는 취소선. 쿠폰/할인 적용 시 칩 노출 — 🎟️ 소프트먼트/매장 쿠폰 −금액, 🔥 할인 −금액.
  • 취소 행: 연한 적색 배경(#fff7f7), 주문번호 취소선+red, "↺ 취소일시 + 취소/결제 취소" 보조 라인.
  • 표시 개수 셀렉트(10/20/50/100, 기본 20)는 화면 페이징에만 적용 — "💡 화면에만 적용돼요 · 내보내기는 조회된 N건 전체". 하단 페이지네이션 제공.
  • 상세 모달 버튼: [🖨️ 영수증 출력](항상) · 완료+1년 이내 주문은 [취소](red, 확인 안내) · 1년 초과는 [🔒 취소 기간 만료] 비활성.
데이터·연동
  • 상세 모달 내용: 주문번호(id) · 승인번호(makeApproval) · 결제번호 tid(maskTid) · 채널(channel(ch)) · 주문일시(취소 건은 취소 처리 일시 함께).
  • 🧾 주문 메뉴 상세: 항목별 name × qty · 옵션 칩(options[]) · 금액(price*qty), 합계 수량.
  • 금액 블록: 소계(originalAmt) · 쿠폰 할인(couponAmt, couponIssuer = softment=정산 보전 / store=매장 발행·정산 제외) · 할인(discountAmt) · 최종 결제금액(amt). 취소 건은 결제금액 취소선 + 실매출 0원.
  • 결제수단(payLabelFull = 1depth+2depth) · 카드번호(maskCard 마스킹) · 외화(currency, 있을 때).
  • 연동: 메뉴명·옵션·스냅샷 가격은 메뉴판 관리와 연결(주문 시점 unit_price_snapshot 유지). 취소·환불은 매출·정산 화면의 원거래일 차감과 연동(SPEC §1.5).
예외처리·UX문구

5주문 상태 · 취소 유형 정의
기능·목적
  • 각 행/모달의 상태값 의미와 취소 유형(주문 취소 vs 결제 취소) 구분 기준을 정의한다.
  • 운영자가 환불 발생 여부를 한눈에 판단하고, 잘못된 상태 전이를 막기 위함.
동작·상태값
  • 상태 흐름: 접수(accepted=결제 완료·신규) → 조리중(cooking) → 조리완료(ready) → 완료(completed) / 취소(cancelled) (STATES §1.1~1.2).
  • 전이 액션: 조리 시작(accepted→cooking) · 조리 완료(cooking→ready) · 픽업/서빙 완료(ready→completed, 포장은 픽업+30분 경과 시 자동) — staff 가능(STATES §1.4).
  • 포장 예약: 결제 후 보류 상태(status='accepted' + reservation_accepted_at IS NULL)에서 예약 수락 / 예약 거절을 거침. 거절은 자동 환불(STATES §1.3).
  • 표시 pill: 완료=done(blue), 진행중=partial(amber), 취소=cancel/pending(red 계열). 취소 라벨은 유형에 따라 "주문 취소"/"결제 취소".
데이터·연동
  • 구분 기준 = 결제 완료 여부(선불/후불 아님). 판정: Payment.status='paid'이면 결제 취소(환불 발생), 아니면 주문 취소.
  • 주문 취소 = 결제 미완료 주문의 취소 → 환불 대상 없음(예: 후불 매장 주문을 결제 전 취소).
  • 결제 취소 = 결제 완료 주문의 취소·반품 → 카드 환불 발생(영업일 3~5일). 후불이라도 결제 완료 후 손님이 돌아와 환불하면 결제 취소. (index.html의 선불=결제취소/후불=주문취소 라벨링은 단순화이며, 이 결제 완료 여부 기준이 우선.)
  • 취소·환불 건은 실매출 0원, 환불은 refund 테이블에 기록(original_payment_id, refunded_at)되고 원거래(결제)일 매출에서 차감(SPEC §1.5). 부분 환불 시 결제건수 1 유지·환불건수 +1.
  • 상태 전이는 낙관적 락(version 컬럼), 충돌 시 최신 상태 재조회 후 재시도(SPEC §1.7).
예외처리·UX문구
  • 완료된 주문은 취소 불가 → 환불로 처리(STATES §1.2 completed→cancelled 불가).
  • 주문 거절(accepted→cancelled, 자동 환불) 및 결제 취소(환불)는 owner 전용, staff 불가(SPEC §1.6, STATES §1.4). 조리/픽업/예약 수락·거절은 staff 가능.
  • 이미 정산 완료된 일자의 사후 환불은 다음 정산 회차에서 차감(SPEC §1.5, 정산 연동).
  • 낙관적 락 충돌 시 사용자에게 최신 상태 재조회를 안내하고 액션 재시도 유도(능동 톤).
