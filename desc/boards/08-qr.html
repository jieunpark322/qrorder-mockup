📱QR 관리 매장 운영 qr
📍 핀 6개
1화면 개요
기능·목적
  • QR 관리 화면의 최상위 목적 정의 영역: 매장에서 쓰이는 모든 QR(테이블 주문 QR·포장하기 QR·메뉴판 QR)을 한 화면에서 확인하고, 종이로 즉시 임시 출력하거나 디자인이 들어간 정식 QR 스티커를 신청하는 진입점이다.
  • QR은 이 화면에서 직접 만들지 않는다. 테이블 관리의 테이블 목록과 주문 방식(먹고가기/포장하기)·메뉴판 설정을 소스로 삼아 자동 파생되므로, 사장님이 별도로 추가/삭제할 필요가 없다는 점을 일관되게 전달한다.
  • 페이지 헤더 문구: 제목 QR 관리, 부제 "테이블·포장 QR을 즉시 임시 출력하거나, 정식 스티커를 신청할 수 있어요." 크럼브는 매장 운영 › QR 관리.
동작·상태값
  • 화면 진입 시 기본 탭은 state.qrMainTab='code'(QR 코드 현황). 신청 폼 상태 기본값은 qrDesignMode='same', qrReqColor=null, qrReqQty=9(=QR_BATCH_SIZE), qrReqMemo=''.
  • QR 자동 생성 규칙: 테이블 QR=TABLES 항목 수만큼 / 포장 QR=ORDER_CHANNELS.mode가 takeout 또는 both일 때 매장당 1개 / 메뉴판 QR=운영 모드와 무관하게 매장당 항상 1개.
  • 연동 화면 설정이 바뀌면(테이블 관리에서 테이블 추가/삭제, 주문 방식에서 포장 ON/OFF) QR 현황이 즉시 자동 갱신된다. 사장님 측의 수동 재생성 액션은 없다.
  • 상단 안내 배너 노출(현황 탭): "QR은 먹고가기 설정·테이블 설정을 보고 자동 생성돼요. 따로 추가하실 필요가 없어요." + 테이블 관리/주문 방식으로 점프하는 인라인 링크.
데이터·연동
  • 소스 데이터: TABLES[](테이블 관리), ORDER_CHANNELS.mode(주문 방식: dine_in|takeout|both), ZONE_LIST[](구역). 이 화면은 이들을 읽기 전용으로 참조해 QR을 렌더한다.
  • QR 주문 링크 패턴: order.qrorder.co.kr/{storeCode}/{slug}. storeCode는 매장 식별자(예: softcoffee-hapjeong), slug는 테이블명/takeout/menu.
  • 스티커 신청 데이터: QR_REQUESTS[](신청 이력), 상수 QR_PRICE_PER_UNIT=2000(원/장), QR_BATCH_SIZE=9(최소·증감 단위), QR_COLOR_OPTIONS[], QR_EXISTING_DESIGN(초도 발송 시안).
  • 공통 규칙 준수: 영업일 03:00 시작(SPEC §1.1), 신청일/발송일 표기는 영업일 기준. 정렬·검색·페이징은 SPEC §2.6, 토스트·로딩·빈상태·모달·접근성은 spec/08-common.md를 따른다.
예외처리·UX문구
  • 권한(SPEC §1.6): QR 현황 확인·임시 출력은 owner·staff 모두 가능하나, 정식 스티커 신청/취소는 owner 전용. staff에게는 스티커 신청 탭의 신청·취소 버튼을 비활성(또는 미노출)하고 "매장 사장님 계정에서 신청할 수 있어요." 안내.
  • 설정 미비 경계: 테이블이 0개면 테이블 QR 영역은 빈 상태, 포장 OFF면 포장 QR은 미생성 상태(각 핀에서 처리). 메뉴판 QR은 항상 1개 보장.
  • 링크 복사 시 클립보드 미지원 브라우저 대비: navigator.clipboard가 없으면 링크 텍스트를 선택 가능한 형태로 노출해 수동 복사를 안내.
2탭 (QR 코드 현황 / 스티커 신청)
기능·목적
  • 2개 탭 전환 영역: ① 📱 QR 코드 현황(현재 매장의 모든 QR 보기·임시 출력) ② 📦 QR 스티커 신청(디자인 들어간 정식 스티커를 배치로 신청·이력 확인).
  • 임시 출력(가정용/매장 프린터로 즉시 종이 출력)과 정식 스티커(유료 제작·배송)를 명확히 분리해, 급할 땐 임시로 쓰고 정식은 따로 신청하도록 흐름을 나눈다.
동작·상태값
  • 탭 클릭 시 state.qrMainTab을 'code'/'sticker'로 설정 후 renderApp() 재렌더. 활성 탭에 .on 스타일.
  • 📦 QR 스티커 신청 탭 라벨 옆에 진행 건수 배지 노출: QR_REQUESTS.length가 1 이상이면 파란 pill로 건수 표시(0이면 배지 숨김).
  • 신청 정책: 한 번의 신청은 장 단위(배치)로 묶어 진행하며, 한 번 신청 시 색상은 1가지만 선택할 수 있다. 색상을 여러 종류로 받으려면 신청을 나눠 진행.
  • 임시 출력은 즉시 가능(가정용 흑백 프린터 가능), 정식 스티커는 신청→제작→배송 절차를 거친다.
데이터·연동
  • QR_REQUESTS[] 레코드 필드: id:number, qty:number(배치 배수), colorName:string, hex:string, memo:string, reqDate:string(YYYY-MM-DD), status:'신청중'|'제작중'|'완료'|'취소', totalPrice:number.
  • 상태(STATES) 전이: 신청중(접수) → 제작중 → 완료(=배송). 신청중 단계에서만 매장이 취소로 전이 가능.
  • 배지 카운트 소스는 QR_REQUESTS.length(취소 포함 전체 이력 건수). 정렬은 최신 신청일 우선 권장(SPEC §2.6).
예외처리·UX문구
  • 배지: 신청 이력이 없으면 배지를 숨겨 0 노출을 피한다(능동·긍정 톤).
  • 권한 부족(staff): 신청 탭 진입은 허용하되 신청·취소 액션은 비활성화하고 "정식 스티커 신청은 사장님 계정에서 진행할 수 있어요." 안내(SPEC §1.6).
  • 정식 스티커 신청은 결제(유료)를 동반하므로 발주 직전 더블 체크 확인창을 거친다(핀 2 신청 흐름은 핀 1 신청 영역 참조).
3QR 종류 (테이블 / 포장 / 메뉴판)
기능·목적
  • QR 종류 퀵 점프 탭 영역(현황 탭 내): 🪑 테이블 QR·🛍️ 포장하기 QR·📖 메뉴판 QR 3종을 한눈에 보고, 각 종류 섹션으로 페이지 내 스크롤 점프한다.
  • 종류별 현재 생성 개수를 배지로 즉시 파악하게 해, 매장 구성(테이블 수·포장 운영 여부)을 빠르게 확인시킨다.
동작·상태값
  • 탭 항목 클릭 시 해당 섹션 앵커(#qr-section-table / #qr-section-takeout / #qr-section-menu)로 페이지 내 스크롤 이동(다른 화면 전환 아님).
  • 각 탭에 개수 배지: 테이블=TABLES.length, 포장하기=isTakeoutQRActive()?1:0, 메뉴판=1(고정).
  • 포장 OFF 상태면 포장하기 배지는 0으로 표시되고 해당 섹션은 비활성 안내(핀 5 참조).
데이터·연동
  • 탭 구성은 qrQuickTabs()가 생성. 개수 소스: TABLES.length, ORDER_CHANNELS.mode(isTakeoutQRActive()), 메뉴판 상수 1.
  • 배지 수치는 연동 설정 변경 시 재렌더로 실시간 반영(테이블 추가→테이블 배지 증가, 포장 ON→포장 배지 1).
예외처리·UX문구
  • 개수 0 표기 허용 영역(포장 OFF 시 0)이나, 문구는 부정형 대신 "포장하기를 켜면 자동 생성돼요" 식 가능형 유도가 바람직.
  • 앵커 점프 대상 섹션이 빈 상태(테이블 0개 등)여도 점프는 동작하고, 해당 섹션 내부에서 빈 상태 안내를 노출.
4테이블 QR
기능·목적
  • 테이블 QR 섹션: 테이블 관리에서 만든 테이블마다 자동 생성된 주문용 QR·주문 링크를 구역별로 묶어 보여주고, 테이블 단위로 임시 출력·링크 복사를 제공한다.
  • 손님이 테이블 QR을 찍으면 해당 테이블로 식별된 주문 화면(order.qrorder.co.kr/{storeCode}/{테이블명})에 진입하도록 연결한다.
동작·상태값
  • 섹션 헤더: "🪑 테이블 QR · 테이블 관리에서 만든 테이블마다 자동 생성돼요" + 우측 "테이블 관리 →" 링크(setSection('table')).
  • 구역(ZONE_LIST)별 그룹핑: 각 구역 헤더에 "📍 {구역명} {n}개". 테이블이 있는 구역만 렌더하고, 구역 내 테이블은 입력 순서대로 1,2,3… 번호 표기.
  • 테이블 행마다: QR 미리보기 이미지(qrMock(t.id))·테이블명·주문 링크 텍스트 + 버튼 2개. 🖨️ 임시 출력 클릭 → openTempPrintQRModal('table', t.id) 모달. 🔗 링크 복사 클릭 → 클립보드에 주문 링크 복사 후 "링크가 복사됐어요" 알림. 링크 텍스트 자체 클릭으로도 복사된다.
  • 테이블 추가/삭제/이름변경은 이 화면에서 불가 — 모두 테이블 관리에서 수행하면 QR 현황에 자동 반영.
데이터·연동
  • 행 소스: TABLES[] (id:number, name:string, zone:zoneId). 구역 소스: ZONE_LIST[](id, name).
  • 주문 링크: order.qrorder.co.kr/{storeCode}/{t.name}. storeCode는 매장 식별자. 테이블명이 slug로 그대로 쓰이므로 테이블명은 URL-safe 해야 함(테이블 관리 측 검증 책임).
  • QR 이미지는 테이블 id 기반 결정론적 패턴으로 렌더(실제 구현에서는 서버 발급 QR 이미지/엔드포인트로 대체).
예외처리·UX문구
  • 빈 상태: 테이블이 0개면 "테이블을 추가하면 QR이 자동으로 생겨요." 안내 + "테이블 관리 →" 이동 버튼 노출(부정형 회피).
  • 클립보드 미지원/거부 시: 복사 실패를 조용히 흘리지 말고 링크를 선택 가능한 모달이나 인라인으로 노출해 수동 복사 유도.
  • 임시 출력 모달은 흑백 종이 출력 전제임을 명시: "이건 임시 출력용이에요. … 디자인이 들어간 정식 QR 스티커는 따로 신청해 주세요."
  • 권한: owner·staff 모두 현황 확인·임시 출력·링크 복사 가능(스티커 신청만 owner 전용).
5포장하기 QR
기능·목적
  • 포장하기 QR 섹션: 포장 주문 전용 QR을 1개 제공한다. 카운터·입구·외부 창구 등 여러 위치에 같은 QR을 출력·부착해 손님이 포장 주문 화면(/{storeCode}/takeout)으로 들어오게 한다.
  • 포장 QR의 존재 여부는 주문 방식 설정(포장하기 ON/OFF)에 종속됨을 명확히 보여, QR을 따로 켜고 끄는 개념이 아님을 전달.
동작·상태값
  • 활성 조건: isTakeoutQRActive() = ORDER_CHANNELS.mode ∈ {takeout, both}. 활성 시 "포장하기 [자동 생성]" 배지와 함께 QR·링크·🖨️ 임시 출력 버튼(openTempPrintQRModal('takeout')) 노출.
  • 활성 헤더 부제: "주문 방식에서 포장하기가 켜져 있어 자동 생성돼요". 보조 문구: "카운터·입구·외부 창구 등 여러 위치에 같은 QR을 출력해서 부착할 수 있어요."
  • 비활성(포장 OFF) 시: 헤더 부제 "포장하기가 꺼져 있어요" + 본문 안내 "현재 매장은 포장하기를 받지 않아요. 주문 방식에서 먹고가기+포장하기 또는 포장하기만으로 바꾸시면 포장 QR이 자동 생성돼요." + "🛒 주문 방식 설정으로 이동" 버튼(setSection('order-type')).
  • 우측 헤더 "주문 방식 →" 링크로 항상 설정 화면 점프 가능.
데이터·연동
  • 활성 판정 소스: ORDER_CHANNELS.mode. 링크: order.qrorder.co.kr/{storeCode}/takeout(고정 slug takeout). QR 미리보기는 qrMock(1000) 식 고정 식별자.
  • 포장 QR은 매장당 항상 1개(테이블처럼 N개 생성되지 않음). 위치별로는 동일 QR을 복수 출력해 사용.
예외처리·UX문구
  • 비활성 빈 상태는 부정형("안 받아요")보다 "포장하기를 켜면 포장 QR이 자동 생성돼요" 가능형 유도가 톤에 부합.
  • 주문 방식에서 포장하기를 끄면 손님 화면에서 포장 QR이 사라진다 — 설정 변경 시 영향 안내("포장하기 QR이 손님 화면에서 사라져요.")와 일관.
  • 권한: 현황·임시 출력은 owner·staff 모두 가능.
6메뉴판 QR
기능·목적
  • 메뉴판 QR 섹션: 주문·결제 없이 메뉴 정보만 보여주는 열람 전용 QR을 매장당 1개 제공한다. 외부 입간판·메뉴 진열대·SNS 공유 등 매장 밖 노출 용도.
  • 운영 모드(먹고가기/포장)와 무관하게 항상 존재하므로, 주문 채널과 분리된 보조 노출 수단으로 정의한다.
동작·상태값
  • 항상 생성: 운영 모드와 무관하게 매장당 1개. 헤더 부제 "매장 운영 모드와 무관하게 자동 생성돼요 · 외부 진열·SNS 공유용".
  • 상단 보라색 안내 배너: "매장 운영 모드와 무관하게 항상 주문/결제가 불가능해요. 손님은 메뉴 정보만 볼 수 있어요. 외부 입간판·메뉴 진열대·SNS 공유 용도로 쓰면 편해요."
  • 행: "메뉴판 (외부 노출용) [메뉴 전용]" 배지 + 링크 + 🖨️ 임시 출력 버튼(openTempPrintQRModal('menu'), isMenuOnly=true). 임시 출력 모달 안내: "메뉴판 전용 QR이에요. 손님은 메뉴만 볼 수 있고 주문/결제 화면은 나오지 않아요."
  • 링크 텍스트/복사로 /menu URL 클립보드 복사 가능.
데이터·연동
  • 링크: order.qrorder.co.kr/{storeCode}/menu(고정 slug menu). QR 미리보기 qrMock(2000) 식 고정 식별자.
  • 노출 메뉴 소스는 메뉴 관리(spec/02-menu.md)의 공개 메뉴 데이터. 손님 화면은 열람 전용 모드로 장바구니·결제 UI를 렌더하지 않는다.
  • 다른 QR과 달리 테이블/주문 채널에 종속되지 않는 독립 자동 생성 항목.
예외처리·UX문구
  • 메뉴가 0개여도 QR 자체는 생성되며, 손님 화면에서는 "준비 중인 메뉴예요" 식 빈 상태 노출(주문 버튼은 어떤 경우에도 비노출).
  • 손님이 메뉴판 QR로 들어와 주문을 시도할 수 없도록, 주문/결제 진입 경로를 전 구간 차단(보안·UX 경계).
  • 권한: 현황·임시 출력은 owner·staff 모두 가능.
