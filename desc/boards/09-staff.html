9
👥직원 계정 관리 매장 운영 staff
📍 핀 6개
1화면 개요
기능·목적
  • 직원 계정 관리 화면 전체의 목적 정의. 매장에서 실제 주문 처리를 하는 직원에게 로그인 가능한 계정을 발급·관리하는 owner 전용 운영 화면.
  • 사장님(owner) 본인 계정은 이 화면에서 다루지 않고 사이드바 상단 사용자 박스 → 내 정보에서 별도 관리한다. 따라서 목록·KPI에서 owner는 항상 제외된다.
  • 직원 계정에는 생성 시 자동으로 'staff' 권한이 부여되며, 권한은 화면에서 선택·변경할 수 없다(단일 권한 모델, SPEC §1.6.1).
동작·상태값
  • 라우트 진입 시 renderStaff(main) 실행. 크럼브 "매장 운영 › 직원 계정 관리", 페이지 제목 "직원 계정 관리".
  • 화면 구성 순서: 툴바(제목+안내문구+[ℹ️ 권한 설명][+ 직원 등록] 버튼) → 안내 박스 → KPI 3카드 → 검색·필터 → 직원 목록 표.
  • owner 전용이므로 staff 역할이 진입하면 화면 자체를 노출하지 않는다(좌측 메뉴는 회색+자물쇠, SPEC §1.6.2 '권한 관리(직원 계정)' staff=✗).
데이터·연동
  • 데이터 소스: STAFF_LIST 배열. 직원 엔터티 필드 = id(number, PK), loginId(string), name(string), phone(string), email(string), groupId(number: 1=사장/owner, 2=직원/staff), useFrom(date|''), useTo(date|''), enabled('Y'|'N'), regDate(date), isMe(bool, owner 본인만 true).
  • 이 화면은 groupId!==1 (사장 제외) 레코드만 다룬다. groupId=2(staff)가 노출 대상.
  • 권한 매트릭스(SPEC §1.6.2)와 연동: 여기서 만든 계정의 역할이 전체 어드민 화면 노출/액션 가능 여부를 결정한다.
예외처리·UX문구
  • owner 전용 화면. staff가 URL 직접 접근 시 권한 안내 모달 "사장님 권한이 필요해요. 가맹점주 본인 계정으로 로그인해 주세요" (spec/08-common.md 403 처리).
  • 데이터 로딩 실패 시 표 영역에 공통 에러 빈 상태 + 재시도. 본인(owner) 계정은 이 화면에 노출되지 않음을 페이지 부제로 안내: "사장님 본인 계정은 사이드바 위 사용자 박스에서 관리할 수 있어요."
2안내 (계정 생성 경로)
기능·목적
  • 매장 계정의 발급 경로를 안내하는 정보 박스. 사장님이 직원 계정과 매장(본사 발급) 계정을 혼동하지 않도록 구분해 준다.
  • 직원 계정 생성 시 자동으로 '직원' 권한이 부여된다는 점과, 상세 권한 범위는 [ℹ️ 권한 설명]에서 확인 가능함을 안내한다.
동작·상태값
  • 표 위에 항상 노출되는 정적 안내 박스(💡 아이콘). 클릭 동작 없음(읽기 전용).
  • 노출 문구: "매장 계정은 가입 시 소프트먼트(본사)에서 만들어 드려요. 사장님은 매장에서 일할 직원 계정만 추가하시면 돼요. 직원에게는 자동으로 '직원' 권한이 부여돼요. 자세한 권한 범위는 우상단 ℹ️ 권한 설명 버튼에서 확인할 수 있어요."
  • [ℹ️ 권한 설명] 버튼 클릭 시 openStaffPermsModal() 실행(핀 6에서 상세).
데이터·연동
  • 정적 텍스트. 바인딩 데이터 없음.
  • 매장(본사 발급) 계정은 groupId=1(owner)로, 가맹 계약 시 소프트먼트가 생성하며 어드민에서 추가/삭제 불가.
예외처리·UX문구
  • 추가 매장 계정이 필요하면 본사 문의 경로(고객 지원)로 안내. 능동·가능형 톤 유지("확인할 수 있어요").
  • 예외 동작 없음(정적 영역). 반응형에서 좁은 화면일 때도 줄바꿈으로 전체 문구 노출 유지.
3계정 현황 KPI
기능·목적
  • 직원 계정 현황을 한눈에 보여 주는 KPI 카드 3종: 전체 직원 계정 / 사용 중 / 사용 안 함.
  • 사장님이 활성 인력 규모와 비활성(퇴사·휴직) 계정 수를 즉시 파악하도록 한다.
동작·상태값
  • 3개 카드 가로 배치(grid-template-columns:repeat(3,1fr)). 클릭 동작 없는 수치 표시 전용.
  • 카드1 "전체 직원 계정" = totalCnt = staff 전체 수. 카드2 "사용 중" = activeCnt(enabled==='Y'), 파란색 강조. 카드3 "사용 안 함" = offCnt(enabled==='N').
  • 각 수치 뒤 단위 "명" 표기. 사용 안 함 카드는 offCnt>0이면 진한 회색, 0이면 연한 회색으로 표시.
  • 수치는 필터·검색과 무관하게 항상 전체 집계(STAFF_LIST.filter(groupId!==1) 기준)로 고정.
데이터·연동
  • 집계식: totalCnt = staff(groupId=2) 전체 건수, activeCnt = enabled==='Y' 건수, offCnt = enabled==='N' 건수. 항상 activeCnt + offCnt = totalCnt.
  • 사용 종료일(useTo)이 지난 계정은 영업일 03:00 경계(SPEC §1.1) 기준으로 자동 enabled='N' 처리되어 '사용 안 함' 집계에 반영된다(핀 6 참조).
예외처리·UX문구
  • 직원 0명이면 세 카드 모두 "0명"으로 표시(빈 상태 텍스트는 표 영역에서 별도 처리).
  • owner 본인 계정은 어떤 카드에도 집계되지 않음.
4필터 · 검색
기능·목적
  • 사용 여부 필터 + 직원 검색으로 목록을 좁히는 영역. 계정 수가 많을 때 원하는 직원을 빠르게 찾도록 한다.
동작·상태값
  • 사용 여부 칩 3종: "전체" / "사용 중" / "사용 안 함". 클릭 시 state.staffUseFilter를 'all'/'Y'/'N'으로 설정하고 renderApp() 재렌더. 기본값 'all'(전체), 활성 칩은 .chip.on 강조.
  • 검색 입력 1개: placeholder "직원명·로그인 ID·휴대폰". oninput으로 state.staffSearch 갱신, onchange(엔터/포커스아웃)에 renderApp() 실행.
  • 검색 매칭 대상: name + ' ' + loginId + ' ' + phone 결합 문자열에 대한 대소문자 무시 부분 일치(SPEC §2.6). 이메일은 검색 대상에서 제외.
  • 필터와 검색은 AND 결합되어 표에 반영. KPI 카드 수치에는 영향을 주지 않음.
데이터·연동
  • state.staffUseFilter: 'all' | 'Y' | 'N' (기본 'all'). state.staffSearch: string (기본 '').
  • 필터링 로직: rows = STAFF_LIST.filter(groupId!==1) → useFilter!=='all'이면 enabled===useFilter → search 있으면 부분 일치(SPEC §2.6 검색 규칙).
  • 정렬·페이징은 SPEC §2.6 공통 규칙 적용 대상이나 현재 표는 등록 순(배열 순) 단일 정렬.
예외처리·UX문구
  • 필터·검색 결과가 0건이면 표 빈 행에 "조건에 맞는 직원이 없어요. 검색·필터를 다시 살펴봐 주세요." 노출(능동·긍정 톤).
  • 전체 직원이 0명이면 "아직 등록한 직원 계정이 없어요. 위 + 직원 등록을 눌러 추가해 주세요." 노출.
  • 검색어 앞뒤 공백은 매칭 시 무시. 빈 검색어면 필터만 적용.
5직원 목록 (표)
기능·목적
  • 직원 계정 목록 표. 등록된 직원의 핵심 정보를 한 줄에 보여 주고, 행에서 바로 수정·삭제·사용여부 확인이 가능하게 한다.
  • 컬럼: # / 직원명 / 로그인 ID / 휴대폰 / 이메일 / 사용 기간 / 사용 여부 / 등록일 / (액션).
동작·상태값
  • 표 헤더 "직원 목록" 옆에 건수 표시: 필터 결과가 전체와 다르면 "N명 · 전체 M명 중", 같으면 "N명".
  • 각 행 셀: # = 표시 순번(1부터), 직원명 = 굵게, 로그인 ID = 모노스페이스, 휴대폰·이메일·등록일은 보조색. 등록일은 YYYY-MM-DD를 YYYY.MM.DD로 표기.
  • 사용 기간 셀: useFrom·useTo가 모두 없으면 "기간 없음"(연한 회색), 있으면 "useFrom ~ useTo" 형식(없는 쪽은 '~').
  • 사용 여부 셀: 상태 pill — 'Y'면 "사용 중"(done/초록), 'N'이면 "사용 안 함"(cancel/회색).
  • 행 액션 셀: [수정] 버튼 → openStaffEditModal(id), [삭제] 버튼(빨강) → deleteStaff(id).
  • 헤더 정렬은 SPEC §2.6 기준으로 확장 가능하나 기본은 등록 순 표시.
데이터·연동
  • 행 바인딩: id, name, loginId, phone, email, useFrom, useTo, enabled, regDate.
  • 검증 규칙(등록/수정 시, SPEC 08 §5.3 폼 검증): 직원 이름 한글·영문 2~10자, 로그인 ID 영문·숫자 5~20자(매장 내 유일), 휴대폰 010-0000-0000 형식, 이메일 정규식 ^[^@\s]+@[^@\s]+\.[^@\s]+$.
  • 등록일(regDate)은 생성 시 서버 영업일(SPEC §1.1, 03:00 시작) 기준 자동 기록.
  • 이메일은 비밀번호 찾기·계정 안내 발송 채널과 연동.
예외처리·UX문구
  • 이메일 미입력 행은 셀에 "—" 표시(현재 모델상 이메일은 필수이나 표시 안전장치).
  • 빈 상태 문구: 전체 0명 "아직 등록한 직원 계정이 없어요. 위 + 직원 등록을 눌러 추가해 주세요." / 필터 0건 "조건에 맞는 직원이 없어요. 검색·필터를 다시 살펴봐 주세요."
  • 로그인 ID·연락처 형식 오류는 저장 단계에서 차단(핀 6 검증 메시지 참조, SPEC 08 §5.3).
  • owner(본인) 행은 목록에 절대 표시되지 않음.
6사용 여부 · 행 액션
기능·목적
  • 계정 사용/중지 전환 및 정보 수정·삭제, 권한 설명, 신규 등록을 담당하는 액션 영역. 계정 사용 여부로 로그인 차단을 제어한다.
  • 포함 요소: [+ 직원 등록] 버튼, [ℹ️ 권한 설명] 모달, 행 [수정]/[삭제], 등록·수정 모달 내 '계정 사용 여부' 토글.
동작·상태값
  • [+ 직원 등록] → openStaffAddModal(): 빈 폼(기본 enabled='Y', groupId=3이나 저장 시 강제 2로 고정), 저장 버튼 "직원 등록하기".
  • [수정] → openStaffEditModal(id): 기존 값 채움, 로그인 ID는 readonly("한 번 정한 ID는 바꿀 수 없어요"), 비밀번호는 빈칸("비워 두면 변경되지 않아요"), 저장 버튼 "저장".
  • 모달 '계정 사용 여부' 토글: ON="사용 중"(enabled='Y'), OFF="사용 안 함"(enabled='N'). 보조문구 "사용 안 함으로 두면 로그인이 막혀요". OFF인 계정은 로그인 차단.
  • [삭제] → deleteStaff(id): confirm "{이름} 직원 계정을 삭제할까요? 이 계정으로는 더 이상 로그인할 수 없게 돼요." 확인 시 목록에서 제거 + 토스트.
  • 사용 종료일(useTo) 경과 시 영업일 03:00 경계(SPEC §1.1)에 자동 enabled='N' 전환("종료일 이후엔 자동으로 사용 안 함이 돼요").
  • 권한은 선택 불가 — 모달에 "자동 '직원' 권한이 부여돼요" 안내만 노출, 저장 시 groupId=2 강제.
데이터·연동
  • 토글 상태 플래그: staff.enabled ('Y'|'N'). 저장 시 등록=STAFF_LIST.push(id=max+1, regDate=오늘), 수정=Object.assign(loginId 제외 갱신).
  • 권한 고정: 저장 시 groupId=2(staff) 강제. 이 값이 SPEC §1.6.2 매트릭스를 통해 전체 어드민 접근 권한을 결정.
  • 비밀번호: 등록 시 필수(영문·숫자·특수문자 8~20자), 수정 시 빈칸이면 미변경. 첫 로그인 시 본인 변경 권장 안내.
  • 사용 여부와 사용 종료일은 로그인 게이트키핑에 함께 사용(enabled='N' 또는 useTo 경과 → 로그인 거부).
예외처리·UX문구
  • 저장 검증 실패 메시지(SPEC 08 §5.3): 이름 미입력 "직원 이름을 입력해 주세요.", 휴대폰 "휴대폰을 입력해 주세요.", 로그인 ID "로그인 ID를 입력해 주세요.", 비밀번호(등록) "비밀번호를 입력해 주세요.", 이메일 미입력 "이메일을 입력해 주세요. 비밀번호 찾기·계정 안내에 필요해요.", 이메일 형식 "이메일 형식을 확인해 주세요. (예: hong@example.com)".
  • 로그인 ID 중복(등록 시) "이미 사용 중인 로그인 ID예요. 다른 ID로 입력해 주세요."
  • 저장 성공 토스트 "✅ {이름} 직원 계정을 등록했어요/수정했어요", 삭제 토스트 "🗑️ {이름} 직원 계정을 삭제했어요".
  • owner(본인) 계정은 이 화면에서 변경/삭제 불가(isMe 가드). 사용여부·정보 변경은 owner만 가능(SPEC §1.6.2, staff=✗).
  • 권한 세분화 요청 안내(권한 모달 하단): "⚠️ 권한 범위를 더 세밀하게 나누고 싶다면 고객 지원 → 서비스 관련 문의로 알려 주세요. 다음 업데이트 때 검토해 드릴게요."
