🔎매출 상세 보기 매출 관리 detail
📍 핀 4개
1개요 · 집계 기준
기능·목적
  • 매출 상세 보기 화면 전체의 진입 헤더·집계 기준 영역. 페이지 제목(매출 상세 보기)·부제·집계 기준 토글·내보내기 버튼을 묶어 제공한다.
  • 이 화면이 다루는 데이터의 시간 범위를 명확히 고지한다: '어제까지 집계된 매출'. 오늘(영업일) 매출은 사전 집계 전이므로 여기서 다루지 않고 주문 내역으로 유도한다.
  • 모든 탭·표·차트가 공통으로 쓰는 매출 집계 기준(자정 기준 / 영업시간 기준)을 한 곳에서 전환하도록 한다.
  • owner 전용 화면 진입점(매출 수치 노출) — staff는 진입 불가(SPEC §1.6.2 매트릭스: 매출 상세 보기 owner ✓ / staff ✗).
동작·상태값
  • 부제 문구: "{탭명} · 어제까지 집계된 매출이에요. 오늘 매출은 주문 내역 › 에서 실시간으로 확인해 주세요." — '주문 내역 ›' 클릭 시 setSection('orders') 로 주문 내역 화면 이동.
  • 집계 기준 토글(2-세그먼트): [🕛 자정 기준](기본, state.useBusinessHours=false) ↔ [🕒 영업시간 기준](true, ON이면 라벨에 (open~close) 표기). 전환 즉시 renderApp() 로 표·차트·KPI 전부 재집계.
  • 토글 ON(영업시간 기준)이면 본문 상단에 안내 배너 노출(bhActiveNotice()). 영업이 자정을 넘기는 매장이면 강조 배너로 '영업 시작 시각부터 다음 날 마감 시각까지를 하나의 영업분으로 묶어 보여준다'고 설명.
  • 토글 우측 ⚙️ 버튼: 영업시간 설정 모달(openBusinessHoursModal()) 오픈.
  • 📤 내보내기 버튼: openExport(15, "매출 상세 · {탭명}", "2026년 5월 1일(금) ~ 5월 15일(금)") 호출 — 현재 탭·기간을 파일명·범위로 전달.
  • 기본 진입 탭은 state.detailTab='period'(날짜별).
데이터·연동
  • 집계 기준 플래그: state.useBusinessHours (boolean, 기본 false=자정 기준). 매장이 영업시간 기준 선택 시 매장별로 영속 저장(라스트 메모리)해 다음 진입에도 유지(예: store_settings.sales_basis). 영업시간 값 businessHoursOpen/businessHoursClose.
  • 데이터 출처(SPEC §2.2): 당일은 결제 트랜잭션 raw를 시간 단위 GROUP BY(실시간), 과거는 영업일 마감(다음 날 02:59:59 직후 배치) 시 사전 집계된 테이블 조회. 매출 상세는 어제까지(사전 집계 완료분)만 다룬다.
  • 영업일 정의(SPEC §1.1): 영업일 03:00 시작 ~ 다음 날 02:59 종료. '오늘/어제'는 영업일 기준.
  • 권한(SPEC §1.6.2): 화면 접근 role=owner 전용. staff 토큰으로는 라우트 진입 차단.
  • 내보내기 메타: 건수·라벨·기간 문자열을 export 모듈에 전달(xlsx 산출).
예외처리·UX문구
  • 오늘 매출은 본 화면 집계 대상이 아님 — 부제에서 "오늘 매출은 주문 내역 › 에서 실시간으로 확인해 주세요."로 능동 유도.
  • staff 권한이 우회 접근 시: 403 처리 후 '이 화면은 사장님 권한으로 볼 수 있어요' 토스트 + 대시보드 리다이렉트(SPEC §1.6).
  • 영업시간 기준 전환 시 자정을 넘기는 매장은 묶임 기준이 헷갈릴 수 있으므로 배너로 "영업 시작 시각(open)부터 다음 날 마감 시각(close)까지를 하나의 영업분으로 묶어서 보여드려요" 강조 안내.
  • 데이터 로딩 중에는 표·차트 영역에 스켈레톤/로딩 표시(spec/08-common.md 로딩 규칙), 조회 실패 시 '잠시 후 다시 시도해 주세요' 에러 + 재시도 버튼.
2분석 탭 (5종)
기능·목적
  • 분석 축을 전환하는 탭 바. 같은 기간 매출을 여러 관점으로 쪼개어 본다.
  • 핀이 정의하는 5종 축: 날짜별 · 시간대별 · 메뉴별 · 메뉴 카테고리별 · 결제수단별. (현재 목업은 여기에 '주문 방식별'을 추가로 구현 — 그린필드 기준 5종을 핵심으로 정의하고 주문 방식별은 확장 탭으로 둔다.)
  • 사장님이 '언제(날짜/시간) · 무엇이(메뉴/카테고리) · 어떻게 결제됐는지(결제수단)' 다각도로 파악하도록 한다.
동작·상태값
  • 탭 클릭 시 setDetailTab(key) → state.detailTab 갱신 후 renderApp(). body(#detail-body)만 해당 렌더 함수로 교체.
  • 탭 라벨: 📅 날짜별(period) / ⏰ 시간대별(hour) / 🥐 메뉴별(menu) / 🗂️ 메뉴 카테고리별(category) / 💳 결제수단별(method). 확장: 🛵 주문 방식별(channel).
  • 시간대별: '오늘 vs 4주 동요일 평균' 막대/라인 차트 + 시간대별 표(건수·매출·4주 평균·증감%·객단가). 기간 콤보는 잠금(오늘 5/15 고정), 변경 안내는 [날짜별] 탭으로 유도.
  • 메뉴별: 판매량·매출·비중·평균단가·환불건수·환불액 표 + 카테고리 칩 필터(전체/커피/베이커리/음료/차·논커피) + '↺ 환불 있는 메뉴만' 필터(detailMenuRefundOnly) + 메뉴명 검색·매출 범위.
  • 카테고리별: 카테고리 표 + 비중 도넛 차트. 행 클릭 시 메뉴별 탭으로 이동하며 해당 카테고리 자동 필터(setDetailCategory).
  • 결제수단별: 비중 도넛 + 결제수단 표(건수·금액·비중·평균 객단가·취소율). 그룹 필터(전체/카드/간편결제/기타). 해외카드는 USD·환율 표기.
데이터·연동
  • 탭 상태: state.detailTab ∈ {period, hour, menu, category, method, channel}, 기본 'period'.
  • 축별 집계 소스: 날짜별=DAY[](일별 enrich), 시간대별=HOUR/HOUR_AVG(4주 동요일 평균), 메뉴별=MENU[](qty/amt/refund/refundCnt), 카테고리=CATEGORY[], 결제수단=METHOD[].
  • 집계 기준 토글(pin1) 적용: 모든 축 데이터에 applyBhFields() 또는 effectiveHourData() 로 영업시간 기준 반영.
  • 탭별 정렬 상태 개별 보존: detailMenuSort·detailCategorySort·detailMethodSort·detailHourSort·detailPeriodSort.
  • 계산식 공통(SPEC §2.1): 비중=항목/합계, 객단가=actual/ocnt, 시간대 증감=(amt-avg)/avg*100.
예외처리·UX문구
  • 시간대별·메뉴별·카테고리별·결제수단별 탭은 기간 콤보가 잠금(고정 기간) 상태 — 기간을 바꾸려면 배너의 "[날짜별] 탭에서 변경해 주세요" 링크로 유도(능동형).
  • 메뉴별에서 '환불 있는 메뉴만' 필터 적용 후 결과가 0개면 빈 상태 — '조건에 맞는 메뉴를 다시 찾아볼 수 있어요'(부정형 회피) + ✕ 초기화 칩 제공.
  • 탭별 빈 데이터(매출 0): 표에 합계만 0으로 표기, 도넛/차트는 '표시할 매출이 아직 없어요' 빈 상태.
  • 도넛·라인 차트는 캔버스 렌더 실패 시에도 표 데이터는 항상 노출(차트는 보조).
3기간 선택
기능·목적
  • 분석 기간을 선택하는 콤보. 일간 / 주간 / 월간 / 기간 선택(custom) 4단위와 날짜 이동(◀▶)을 제공.
  • 선택 기간을 영업일 경계(03:00~다음 날 02:59) 기준의 실제 조회 구간으로 환산해 표시(startTime/endTime).
  • 날짜별 탭에서 사장님이 '한 날만' 또는 '여러 날 묶음'을 자유롭게 오가도록 한다.
동작·상태값
  • 기간 단위: [일간](day, 기본) / [주간](week) / [월간](month) / [기간 선택](custom). 클릭 시 setDetailGroup(g).
  • 일간 모드: ◀▶ 화살표로 일자 이동(setDetailDayPrev/setDetailDayNext, detailDayIdx 0~13). 첫날(5/1) 도달 시 ◀ 비활성, 어제(5/14) 도달 시 ▶ 비활성. 안내 배너: "한 날만 보기 — ◀▶ 버튼으로 다른 날로 이동할 수 있어요. 여러 날을 한 번에 보려면 기간 선택을 눌러 주세요."
  • 단일 일자일 때(day 또는 custom 시작=종료): '하루 흐름 한눈에 보기' 패널(피크/한산 시간·시간당 평균·결제 건수 KPI + 시간대 막대 차트)로 표시.
  • 주간 모드: 5월 1주차(5/1~5/3)·2주차(5/4~5/10)·3주차(5/11~5/15) 묶음 행. 월간 모드: 1~5월 월별 합계 행.
  • 기간 선택(custom): 시작일·종료일 지정 → 그 사이 일별 데이터 표시, centerLabel에 'M월 D일(요일) ~ M월 D일(요일) · N일간'.
  • 중앙 라벨/구간 시각 동시 표기: 예 일간 시 startTime '5/14(목) 03:00' ~ endTime '5/15(금) 02:59'.
데이터·연동
  • 상태: state.detailGroup ∈ {day, week, month, custom} 기본 'day'; detailDayIdx(기본 13=5/14 어제); custom은 customStart/customEnd(customTarget='detail').
  • 일별 소스 DAY[] + enrichDay()로 gross/disc/pt/ref/actual/taxable/vat/net/ocnt/rcnt 산출. 주/월은 days 합산 후 enrich.
  • 구간 환산: 영업일 03:00 시작 규칙(SPEC §1.1)으로 startTime=해당일 03:00, endTime=다음 영업일 02:59.
  • 이동 한계: 미래(오늘 이후) 이동 불가 — next 비활성. 데이터 보유 범위는 5/1~5/14(어제).
  • 기간 합계(tot)와 객단가(tot.actual/tot.ocnt)는 콤보 선택 범위 기준으로 KPI 카드·표 합계에 반영.
예외처리·UX문구
  • 오늘 이후 날짜로 이동 시도: ▶ 화살표 비활성(disabled) 처리하여 클릭 불가 — 미래 매출은 아직 집계 전.
  • custom 종료일을 어제(5/14)보다 이후로 지정 시: 어제까지로 자동 보정 또는 '어제까지 조회할 수 있어요' 안내.
  • custom 시작>종료 등 잘못된 범위: 폼 검증으로 '시작일은 종료일보다 앞이어야 해요' 인라인 에러(spec/08-common.md 폼검증).
  • 주간/월간 모드에서는 단일 일자 KPI 패널 대신 추이 라인 차트로 전환(데이터 1행일 때만 단일 일자 처리).
4상세 표 · 행 클릭 → 결제 단위 상세
기능·목적
  • 기간별 상세 표 — 행(날짜·주·월·기간)마다 총매출·할인·포인트·환불·실매출·과세금액·부가세·결제건수·환불건수를 한 줄로 보여주는 핵심 데이터 그리드.
  • 행을 클릭하면 그 단위의 결제 단위 상세 모달('📋 OO 매출 상세 내역')로 드릴다운해 개별 결제(주문일시·승인번호·결제수단)를 확인한다.
  • 매출 계산식(총매출 − 할인 − 포인트 − 환불 = 실매출 → 과세금액 + 부가세)을 표 형태로 검증 가능하게 한다.
동작·상태값
  • 표 컬럼: No. / 날짜(주·월·기간) / 총매출 / − 할인 / − 포인트 / − 환불 / = 실매출 / 과세금액 / 부가세 / 결제건수 / 환불건수. 맨 아래 합계 행 고정.
  • 헤더 클릭 정렬: setDetailPeriodSort — 오름↔내림 토글, 활성 컬럼에 화살표(SPEC §2.6). 기본 {key:'d', dir:'desc'}(최신일 우선).
  • 행 클릭: "{라벨} 자세히 ›" → openPeriodDetail(label, date) 모달. title 툴팁 "이 날(기간)의 결제 내역을 볼 수 있어요".
  • 모달 내용: 상단 칩 [결제완료 N건] · [합계 X원](강조) · [취소 M건]. 표 컬럼=주문일시 / 주문번호 / 승인번호 / 메뉴 / 판매유형 / 결제수단 / 금액 / 상태(pill). 하단 버튼 [닫기] [📋 주문 내역 전체 보기].
  • 내보내기(xlsx): pin1의 📤 버튼이 현재 표를 export. 표시 개수 셀렉트(detailRowsPerPage, 10/20/50/100, 기본 목업 50→SPEC 기본 20) + 페이지네이션.
  • 취소 행 표시: 모달에서 취소건은 분홍 배경·주문번호 빨간 취소선·금액 취소선, '↺ M/D HH:MM 취소 처리' 표기 — 결제완료 집계·합계에서 제외.
데이터·연동
  • 행 필드(SPEC §2.1): gross, disc, pt, ref, actual=gross−disc−pt−ref, taxable=actual, vat=round(taxable/11), net=actual−vat, ocnt=count(payment status='paid'), rcnt=count(refund)(부분환불 포함).
  • 모달 결제 행: ORDERS[]에서 해당 일자(o.date) 필터. 필드 date/time/id/makeApproval(o)(승인번호)/menu/channel(판매유형)/payLabel(o)/amt/status.
  • 취소 구분(제품 규칙 우선): 환불·취소 구분은 선불/후불이 아니라 Payment.status==='paid' 여부로 판정 — paid면 결제 취소(환불 발생), 아니면 주문 취소. 후불이라도 결제 완료 후 환불하면 결제 취소.
  • 환불 차감(SPEC §1.5): 환불은 발생일이 아니라 원거래(결제)일 매출에서 차감(refunds.original_paid_at 영업일 기준). 정산 완료분 사후 환불은 다음 정산 회차에서 차감.
  • 연동: 행→openPeriodDetail 모달, 모달 [주문 내역 전체 보기]→setSection('orders'). 카테고리 행→메뉴별 탭 자동 필터.
  • 합계 행: 표시 중인 모든 행을 컬럼별 누계, 객단가=tot.actual/tot.ocnt.
예외처리·UX문구
