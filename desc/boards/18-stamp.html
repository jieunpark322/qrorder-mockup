⭐스탬프·포인트 마케팅·CRM stamp
📍 핀 4개
1개요 · 저장
기능·목적
  • 화면 최상단 크럼브 + 타이틀 영역이자 화면 전체의 저장 컨트롤러. 크럼브 "마케팅 · CRM › 스탬프·포인트", 타이틀 "스탬프·포인트", 서브카피 "손님이 모은 스탬프와 받은 리워드를 관리해요."를 노출한다.
  • 이 화면은 GLOBAL_SAVE_SECTIONS에 포함(index.html:4610)되어 본문 하단에 전역 저장 바가 자동으로 붙는다. 핀2 적립 규칙의 변경을 모아 한 번에 저장하는 단일 저장 지점이다.
  • 마케팅·CRM 그룹 화면으로 쿠폰·멤버십·리뷰와 함께 사장님 전용 CRM 진입점 역할을 한다.
동작·상태값
  • 화면 진입 시 현재 적립 규칙(rule)을 saved snapshot으로 저장한다(SPEC §2.8). 입력값이 snapshot과 달라지면 dirty=true가 되고 하단 저장 바가 활성화된다.
  • [저장] 버튼: dirty일 때만 활성(§08-common 5.4). 클릭 시 버튼 텍스트 "처리 중..." + 인라인 스피너, 성공 시 success 토스트 "변경 사항을 저장했어요" 노출 후 snapshot 갱신·dirty 해제. 실패 시 error 토스트 + 입력값 유지.
  • [되돌리기/취소] 버튼: dirty 상태에서 노출. 클릭 시 입력값을 snapshot으로 롤백하고 dirty 해제.
  • 저장 바 기본 상태: dirty=false면 저장 바 숨김 또는 비활성. 다른 섹션으로 이동 시 dirty면 확인 prompt "변경 사항을 버리고 이동할까요?" 노출(§2.8). ESC·배경 클릭에도 동일 가드 적용(§08-common 6.2).
  • 크럼브 각 단계는 클릭 시 해당 상위 화면으로 이동(§08-common 8.3).
데이터·연동
  • 바인딩 소스: STAMPS.rule (index.html:9736). 저장 단위는 매장 1개 단위가 아닌 전 매장 공통(핀2 참조).
  • 신규 데이터 모델 제안: STAMP_RULE 테이블 — brand_id(전 매장 공통이므로 매장이 아닌 브랜드/사장 단위 FK), stamps_per_order int, reward_at int, reward_type text, updated_at timestamptz, updated_by(owner user id).
  • 저장 API 성격: PUT /stamp/rule (멱등, 전체 규칙 객체 교체). 200ms 이하 응답은 로딩 표시 생략(§08-common 2.2).
  • dirty 플래그는 클라이언트 상태(state)로만 관리, 서버 미전송.
예외처리·UX문구
  • 권한: 기본값 사장님 전용. 권한은 추후 메뉴·카테고리별로 설정(직원 노출 여부 지정)하는 기능으로 관리 예정. 권한 없는 항목은 좌측 메뉴 회색+자물쇠(클릭 시 403) — SPEC §1.6.2 원칙. 직접 URL 진입 시 403 모달 "사장님 권한이 필요해요. 가맹점주 본인 계정으로 로그인해 주세요"(§08-common 4.1).
  • 세션 만료(401): 로그인 화면 리다이렉트 + "다시 로그인이 필요해요".
  • 저장 실패(5xx): error 토스트 + "다시 시도" 버튼, 폼 값 유지. 5회 연속 실패 시 본사 자동 알림(§08-common 4.1).
  • 네트워크 오프라인: 상단 노란 배너 "네트워크 연결을 확인해 주세요", 자동 재시도 3회.
2적립 규칙
기능·목적
  • 좌측 패널 "적립 규칙 <span>전 매장 공통</span>". 손님이 주문할 때 스탬프가 얼마나 적립되고 몇 개를 모으면 어떤 리워드를 받는지를 사장님이 설정하는 영역이다.
  • 하단 안내 배너로 현재 규칙을 자연어로 요약해 사장님이 설정 결과를 즉시 확인할 수 있게 한다.
  • 여기서 정한 규칙이 핀3·핀4의 진행률(stamps/rewardAt) 계산과 리워드 발급 판정의 기준이 된다.
동작·상태값
  • 주문 1건당 적립 스탬프 (number input, 기본값 1): 손님 결제 완료 주문 1건당 적립되는 스탬프 개수. 입력 변경 시 dirty.
  • 리워드 지급 기준 (number input, 기본값 10): 누적 스탬프가 이 값에 도달하면 리워드 1회가 발급된다. 도달 시 스탬프가 차감(stamps -= rewardAt)되고 rewards += 1되는 순환 적립 방식.
  • 지급 리워드 (text input, 기본값 "무료 음료 1잔", grid 전체 폭): 손님에게 지급되는 보상 설명 문구.
  • 안내 배너: 입력값에 실시간 연동되어 "💡 현재 규칙: 주문 N건마다 스탬프 1개, M개 모이면 {리워드} 지급."으로 갱신.
  • 규칙 변경은 저장 이후 발생하는 적립부터 적용되며 기존 손님의 누적 스탬프(핀3)는 소급 변경하지 않는다(description.html 핀2 예외).
데이터·연동
  • 필드: stampsPerStamp(=주문 1건당 스탬프, int ≥1), rewardAt(int ≥1), rewardType(string 1~30자).
  • 적립 트리거: 주문의 Payment.status=paid가 되는 시점에 스탬프 적립. 주문 취소 vs 결제 취소는 선불/후불이 아니라 결제 완료 여부로 구분 — 결제 완료(paid) 후 환불(결제 취소)이 발생하면 해당 주문으로 적립된 스탬프도 회수(stamps -= stampsPerStamp)해야 한다. 결제 전 단순 주문 취소는 애초에 적립되지 않았으므로 회수 없음.
  • 환불 차감은 원거래일 매출 기준(SPEC §1.5)과 동일 사상 — 스탬프 회수도 적립을 유발한 원주문에 귀속.
  • 검증: stampsPerStamp·rewardAt는 0 초과 정수(§08-common 5.3 가격 규칙 준용), rewardType은 빈 값 불가·1~30자. 연동 화면: 핀3 진행률·핀4 행 계산식의 분모로 사용.
예외처리·UX문구
  • 입력 검증 실패: 필드 테두리 빨강 + 필드 아래 빨간 메시지(§08-common 5.2). 예) rewardAt에 0 입력 시 "1 이상으로 입력해 주세요". 검증 실패 시 저장 버튼 비활성.
  • 경계값: rewardAt=1이면 매 적립마다 리워드 발급되는 정상 케이스(허용). stampsPerStamp가 rewardAt보다 큰 경우도 허용하되 안내 배너로 결과를 그대로 보여 준다.
  • 리워드 문구 길이 초과: 30자 초과 시 "30자 이내로 입력할 수 있어요".
  • 규칙 변경 시 기존 적립 소급 안내(필요 시 info 문구): "변경한 규칙은 저장 이후 적립부터 적용돼요".
3손님별 스탬프
기능·목적
  • 우측 상단 "스탬프 현황" KPI 카드 2개와 하단 "회원별 스탬프" 표 헤더를 포함하는, 적립 현황 요약·목록 영역이다.
  • 사장님이 스탬프를 보유한 손님 규모와 누적 지급된 리워드 규모를 한눈에 보고, 손님별 적립 상세를 목록으로 탐색하게 한다.
동작·상태값
  • KPI 카드 — 스탬프 보유 회원: members.length 명. 리워드 지급 누계(파란 카드): members.reduce((a,m)=>a+m.rewards,0)로 전체 회원의 리워드 받은 횟수 합계.
  • 표 헤더: 손님 | 연락처 | 스탬프 | 리워드 받은 횟수(우측정렬) | (상세 버튼 열). 헤더 우측 "<span>N명</span>" 카운트 표시.
  • 헤더 클릭 정렬(description.html 핀3 + SPEC §2.6): 손님명·스탬프 수·리워드 횟수 컬럼 클릭 시 오름차순↔내림차순 토글, 활성 컬럼에 화살표 표시. 기본 정렬은 스탬프 수 내림차순 권장.
  • 검색: 손님명·연락처 부분 일치(대소문자 무시, §2.6). 목록 30건 초과 시 페이징/무한스크롤 적용(§2.6).
  • 각 행의 진행률·리워드 가능 배지는 핀4에서 렌더(핀2 규칙 기준).
데이터·연동
  • 바인딩: STAMPS.members[] = {id, name, phone, stamps, rewards} (index.html:9738~9741).
  • 신규 데이터 모델 제안: STAMP_MEMBER — id, brand_id, customer_id(손님 식별, 연락처 해시 기반), name, phone, stamps int default 0, rewards int default 0, updated_at. 손님은 결제 시 입력/연동된 연락처로 식별·자동 생성된다.
  • 조회 API: GET /stamp/members?sort=&q=&page= (정렬·검색·페이징 쿼리). KPI는 동일 응답의 집계 필드 또는 별도 GET /stamp/summary.
  • 계산식: 보유 손님 수 = 행 수, 리워드 누계 = Σrewards. 표는 단순 조회이므로 토스트 미사용(§08-common 1.3).
예외처리·UX문구
  • 빈 상태(손님 0명): 표 영역에 빈 상태 카드 — "아직 스탬프를 모은 손님이 없어요" + 가능형 보조문구 "손님이 주문하면 자동으로 적립돼요"(§08-common 3.2, 부정 표현 회피). KPI 카드는 0명/0으로 정상 표시.
  • 개인정보 마스킹: 연락처는 모노스페이스로 노출되되 손님명은 "홍**"처럼 마스킹 표기(description.html 핀3 예외). 연락처도 정책에 따라 가운데 자리 마스킹 권장(예 010-****-5678).
  • 검색 결과 0건: "검색 결과가 없어요. 다른 이름이나 번호로 찾아 주세요".
  • 모바일: 표 가로 스크롤 + 힌트 "↔️ 좌우로 스크롤해서 모든 정보를 확인할 수 있어요"(§08-common 10.2).
4손님 행
기능·목적
  • "회원별 스탬프" 표의 개별 회원 행. 한 회원의 적립 진행 상태와 리워드 수령 이력을 시각적으로 보여 주고 상세 진입을 제공한다.
동작·상태값
  • 셀 구성: 손님명(bold) / 연락처(모노스페이스) / 스탬프 진행 셀("{stamps}/{rewardAt}개 · {pct}%" + 진행 바) / 리워드 받은 횟수("{rewards}회", 우측정렬) / [상세] 버튼.
  • 진행률: pct = round(stamps/rewardAt*100), 진행 바 폭은 min(100, pct)%. stamps >= rewardAt이면 파란색 "🎉 리워드 가능" 배지 표시.
  • [상세] 버튼(btn-ghost): 클릭 시 해당 손님의 스탬프 적립/리워드 내역 상세를 연다. 현재 목업은 alert('{name} 스탬프 내역 (와이어프레임)') 자리표시자이며, 정식 구현은 적립·차감·리워드 발급 타임라인 모달(form/info 모달, §08-common 6)로 구현한다.
  • 행 클릭 시 상세(description.html 핀4): 행 전체 클릭으로도 상세 모달 진입 가능(상세 버튼과 동일 동작).
데이터·연동
  • 필드: m.name, m.phone, m.stamps(=stampCount), m.rewards(=rewardCount) (description.html 핀4 data).
  • 상세 모달 데이터: STAMP_LEDGER 제안 — member_id, delta int(+적립/−회수/−리워드차감), reason enum(earn/refund_revoke/reward_issue), order_id(연동), occurred_at. 적립/회수는 결제 완료·환불(결제 취소) 이벤트와 연동.
  • 상세 API: GET /stamp/members/{id}/ledger.
  • 리워드 발급은 stamps>=rewardAt 도달 시 자동(핀2 순환 차감)으로 rewards 증가.
예외처리·UX문구
  • 경계값 — rewardAt 도달/초과: stamps>=rewardAt일 때 "🎉 리워드 가능" 배지 노출, 진행 바는 100%로 클램프. 다회 적립으로 pct>100이어도 바는 100% 고정.
  • 리워드 0회 손님: "0회"로 정상 표시(부정 문구 미사용).
  • 상세 내역 없음: 모달 빈 상태 "아직 적립 내역이 없어요".
  • 결제 취소(환불) 반영: 환불로 스탬프 회수 시 진행률·배지가 즉시 갱신되며, 회수로 stamps가 음수가 되지 않도록 0으로 하한 처리(max(0, stamps)).
  • 접근성: 상세 버튼에 aria-label "{손님명} 스탬프 내역 보기", 진행률은 색만이 아니라 텍스트(N/M개·%) 병행(§08-common 7.1·7.3).
