04 · 옵션 최대 선택 개수 (양방향)

메뉴×옵션 그룹×항목 조합으로 「손님이 이 항목을 몇 번까지 추가할 수 있는지」를 설정. 메뉴 측·옵션 측 어디서나 편집 가능하고 동일 데이터를 공유. (예: 「샷 추가」 — 아메리카노 3회, 디카페인 라떼 1회, 키즈 음료 0회)

📍 라이브 보기 진입 경로 A — 메뉴 관리 → 메뉴판 → 메뉴 클릭 → 판매 설정 → 🎚️ 옵션 최대 선택 개수 진입 경로 B — 메뉴 관리 → 옵션 그룹 → ✏️ 수정 → 🎚️ 메뉴별 최대 선택 개수
📷 화면 캡처 · 옵션 그룹 수정 모달의 「🎚️ 메뉴별 최대 선택 개수」 섹션 — 메뉴 측·옵션 측 양방향 편집
변경 전
옵션 항목은 그룹 단위로만 정의됨. 메뉴별로 「샷 추가는 아메리카노에서만 3회까지」처럼 메뉴마다 다른 한도를 줄 수 없었음.
변경 후
메뉴 측·옵션 측 두 곳에서 같은 데이터를 편집. 옵션 그룹은 항목 정의(이름·금액)만 담당, 「몇 번 추가 가능한지」는 메뉴 측 데이터로 분리.

주요 요소

1
메뉴 측 진입 — 🎚️ 옵션 최대 선택 개수
메뉴 수정 모달 · 섹션 4 (판매 설정)
🎯 기능·목적
  • 현재 메뉴에 연결된 옵션 그룹들의 각 항목 한도를 한눈에 설정
  • 이 메뉴만의 한도를 정의 (다른 메뉴에 영향 없음)
  • 옵션 그룹이 없으면 보라색 안내 카드 + 「옵션 그룹 →」 진입 버튼
⚙️ 동작·상태값
  • 연결된 옵션 그룹별로 카드 표시 (그룹명 · 모드 · 항목 리스트)
  • 각 항목 옆에 input type=number (min=1, max=10) — [N] 회까지
  • 입력 변경 시 setMenuOptMaxQty(groupId, itemIdx, val) 호출 → state._menuEdit.optMaxQty 갱신
  • 모달 저장 시 m.optMaxQty로 영구 반영
  • 1개만 선택 모드는 항목별 1회 고정 (입력 없이 안내 메시지만)
🔌 데이터·연동
  • MENUS[id].optMaxQty: { [groupId]: { [itemIdx]: number } }
  • 예) { '1': {'0':3,'1':1,'2':2} } = 옵션 그룹1의 0번 항목 3회, 1번 1회, 2번 2회
  • 옵션 그룹 정의: OPT_GROUPS[id].items[itemIdx] (이름·금액만, 횟수 없음)
  • 기본값: 데이터 없으면 1회로 표시
⚠️ 예외·UX 문구
  • 연결 옵션 그룹 0개 → "🧩 옵션 그룹이 연결돼 있지 않아요." + 옵션 페이지 진입 버튼
  • 1개만 모드 — "1개만 선택 · 항목별 1회 고정" 회색 라벨
  • 입력 범위 벗어남 → Math.max(1, Math.min(10, ...)) 자동 보정
  • 안내: "메뉴마다 옵션 한도를 다르게 정할 수 있어요. 예) 샷 추가 — 아메리카노 3회, 디카페인 라떼 1회, 키즈 음료 0회"
2
옵션 측 진입 — 🎚️ 메뉴별 최대 선택 개수
옵션 그룹 수정 모달 · 본문 하단
🎯 기능·목적
  • 현재 옵션 그룹이 연결된 모든 메뉴의 항목 한도를 한 화면에서 일괄 관리
  • 옵션 1개를 N개 메뉴에 적용했을 때 한꺼번에 조정 (탭 전환 없이)
  • 각 메뉴 카드에 「메뉴 정보 수정 →」 링크로 메뉴 모달로 빠르게 전환
⚙️ 동작·상태값
  • 옵션 그룹의 links[]에 있는 모든 메뉴를 카드로 나열
  • 각 메뉴 카드 — 메뉴명 + 옵션 항목별 횟수 input
  • 입력 변경 시 setOptGroupMenuMaxQty(menuId, groupId, itemIdx, val) → MENUS[menuId].optMaxQty 즉시 반영 (저장 버튼 불필요)
  • 1개만 모드면 카드 대신 안내 박스만
🔌 데이터·연동
  • 대상 데이터: 메뉴 측과 완전 동일 — MENUS[menuId].optMaxQty[groupId][itemIdx]
  • 입력 함수: setOptGroupMenuMaxQty(menuId, groupId, itemIdx, val)
  • ensureMenuExtras(m) 호출로 옛 메뉴 데이터 자동 초기화
  • markUnsavedUI() 호출로 상단 저장 알림 표시
⚠️ 예외·UX 문구
  • 연결 메뉴 0개 — "🔗 아직 이 옵션 그룹을 적용한 메뉴가 없어요." 안내
  • 1개만 모드 — "🎚️ 메뉴별 최대 선택 개수 — 1개만 선택 모드라 항목별로 1회씩 고정돼요."
  • 「메뉴 정보 수정 →」 클릭 — 옵션 모달 닫고 메뉴 모달 열기
  • 안내: "같은 옵션 그룹이라도 메뉴마다 다르게 정할 수 있어요."
3
양방향 데이터 일관성
메뉴 측 입력 ↔ 옵션 측 입력 동기화
🎯 기능·목적
  • 옵션 측에서 바꿔도 → 메뉴 측 표시값 자동 동일
  • 메뉴 측에서 바꿔도 → 옵션 측 표시값 자동 동일
  • 한 데이터 = 한 진실 원칙 (Single Source of Truth)
⚙️ 동작·상태값
  • 옵션 측: MENUS[menuId].optMaxQty 즉시 갱신 (저장 버튼 없음)
  • 메뉴 측: state._menuEdit.optMaxQty 임시 → 저장 시 MENUS 반영
  • 모달을 새로 열 때마다 ensureMenuExtras로 최신 데이터 로드
🔌 데이터·연동
  • 저장 위치 (단일): MENUS[menuId].optMaxQty
  • 읽기 위치: 메뉴 모달·옵션 모달·손님 화면 모두 같은 필드 참조
  • 구조: { groupId: { itemIdx: maxQty } }
  • 키는 항상 문자열 (String(groupId))
⚠️ 예외·UX 문구
  • 메뉴 모달에서 옵션 측을 통해 값이 바뀌어도 — 메뉴 모달 다시 열면 동일 값 표시
  • 옵션 측 변경은 즉시 반영 — 메뉴 모달에서 「변경하지 않음」 의도 없음 가정
  • 메뉴 모달은 「취소」 시 변경 사항 폐기 (옵션 측에서 바꾼 값은 이미 저장됨)
  • 안내: "메뉴 정보 수정과 같은 데이터" 명시
설계 결정 — 옵션 측은 즉시 저장(메뉴 측 데이터 직접 수정)으로 가는 게 「양방향 편집」의 본래 의도에 부합해요. 두 곳에서 「저장」 버튼이 따로 있으면 같은 값을 두 번 저장해야 하는 혼란이 생깁니다. 메뉴 모달의 「취소」는 메뉴 자체 속성(이름·가격 등)에만 적용되고, 옵션 측에서 이미 즉시 저장된 한도는 유지돼요.
