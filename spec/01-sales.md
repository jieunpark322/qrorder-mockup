# 매출 · 정산 정책

`SPEC.md` §1.5(환불)·§2(매출)와 `DATA_MODEL.md` §8(Settlement), `STATES.md` §8(정산)의 정책 출처 문서. 금액·집계·정산의 단일 기준을 정의한다.

---

## 1. 매출 집계

- **실매출(actual)** = `gross − disc − pt − ref` (총매출 − 할인 − 포인트 사용 − 환불). — SPEC §2.1
- **부가세** = `round(taxable / 11)` (과세금액=공급대가 기준). 간이과세 매장은 부가세 0. — SPEC §1.3
- **영업일 경계** = 03:00 시작 ~ 다음 날 02:59 종료. 자정 이후 거래도 영업 시작일 매출로 묶는다. — SPEC §1.1
- **집계 표시 기준 토글** = 기본 `자정 기준`. 매장이 `영업시간 기준`으로 바꾸면 그 선택을 매장별로 저장(라스트 메모리)해 다음에도 유지한다.

## 2. 환불

- 환불은 **원거래(결제)일의 영업일 매출에서 차감**한다(환불 발생일이 아님). — SPEC §1.5 / DATA_MODEL §4.5
- **주문 취소 vs 결제 취소**는 결제 완료 여부로 구분: `Payment.status='paid'`면 **결제 취소**(환불 발생, 카드 환불 영업일 3~5일), 아니면 **주문 취소**(환불 없음). 후불이라도 결제 완료 후 환불은 결제 취소.
- 부분 환불 시 결제건수는 유지하고 환불건수만 +1.

## 3. 쿠폰과 매출/정산

- **매장 발행 쿠폰**(`coupon_issuer='merchant'`): 사장님 할인(`disc`)으로 **실매출에서 차감**. 정산 보전 없음(매장 부담).
- **소프트먼트 발행 쿠폰**(`coupon_issuer='softment'`): 손님 할인분을 **정산 시 매장에 보전**(`Settlement.platform_coupon_comp`, +).

## 4. 정산

- **정산 주기**: 영업일 종료 후 **D+3 영업일** 출금(공휴일·주말은 영업일 계산에서 제외). — STATES §8 / DATA_MODEL §8.1
- **받으실 금액(net_payable)** = `gross − refund − pg_fee + platform_coupon_comp`.
  - `gross` 해당 영업일 결제 합계 / `refund` 해당일 결제분 환불(사후 환불 포함) / `pg_fee` PG 수수료 / `platform_coupon_comp` 소프트먼트 쿠폰 보전(+).
- **정산 상태**: `정산대기(pending, '곧 들어와요')` → `정산완료(paid_out, '입금 완료')`. 예외로 `보류(held, '확인 중')`.
- **마감 후 환불**: 원거래 정산이 이미 `paid_out`된 뒤 환불이 발생하면 **다음 정산 회차에서 차감**. 다음 회차 정산액이 환불액보다 작으면 `held`(보류) 처리 후 본사 CS 채널로 후속 처리. — STATES §8.2
- **표시 단위**: 정산 내역은 1영업일 = 1행. 토·일·공휴일 등은 한 정산일에 여러 영업일이 묶일 수 있다(회차/settlementId 기준).

## 5. 권한

- 매출 수치·정산은 기본 **사장님 전용**(직원 비노출). 권한은 추후 **메뉴·카테고리별 설정 기능**으로 사장님이 직원 접근 여부를 지정한다. — SPEC §1.6
